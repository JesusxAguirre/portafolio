Weather APP

About the idea

The idea of the project was to consume a weather application using the API from the website https://openweathermap.org/api, with the aim of testing different back-end and front-end skills, and most importantly, practicing the consumption of APIs, which is a commonly used feature in web development.

The challenge

The challenge was to design the database in a way that would store the city name parameter, as it is used to make the request to the API. Additionally, the project involved using AJAX methods for asynchronous requests and implementing backend validations to handle invalid cases and provide appropriate responses to the user.

The solution

The solution involved creating a simple table with a unique ID field and another field called "name," using the SQLAlchemy ORM. The project follows the MVC (Model-View-Controller) design pattern, where only objects of the Weather class interact. In the controller, most exceptions are validated, including data type validations, and most importantly, the validation of the existence of the city. In case of an error, an error blueprint called "errors.py" captures all the application errors and returns JSON error messages to the browser with an appropriate HTTP status code, as well as to the AJAX request.


The rest of the mechanisms are straightforward. When displaying the main page, all rows from the "weathers" table are first queried. Then, for each row, the "name" property is extracted, and it is sent as a URL parameter to the consumed API. The response is stored in a dictionary, which is then added to a list. After looping through each row in the table, this list, containing dictionaries, is sent to the Jinja template. The Jinja "for" statement is used to iterate through the list and display the data accordingly..

The solution

The most important part of this project is the security of information and access to different system pages. Although I would like to go into detail about the mechanisms used, it might be a bit lengthy, so if you want to delve into it, you can find the project on my GitHub. To summarize, I used Flask-Login, a Flask plugin, for user session management. I utilized its decorator for routes that required logged-in users. Additionally, I verified the user's role at the beginning of each route. You can review this in the code.


For this purpose, I created a user table with basic user data and a role field. Although I considered using a relational table, I wanted to keep the project simple and not make it more complex than necessary.


Similar to the previous project, I followed the MVC design pattern, adhering to the theory of abstraction layers, where only user objects are handled in the back-end. The majority of validations are performed in the controller using an additional validation module called helpers.py. It contains various validations, ranging from regular expressions to database checks.


Furthermore, I implemented different asynchronous requests using AJAX, employing various HTTP methods such as GET, POST, UPDATE, and DELETE. I handled different error codes for incorrect requests. For certain errors, such as 404 or 422 for invalid page accesses, I created customizable HTML pages to enhance the user experience.
